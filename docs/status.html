① 設計目的

Symphony 是 openai/symphony SPEC.md 的 Python 實作,圍繞「外部 LLM agent 透過 RESTful Tool API 把 Symphony 當 coding-service backend」這個契約而設計。Dashboard 是用來「觀察」長時間自主運行的 agent,不是用來逐步介入指揮。

Local-first

tracker 預設是內建 SQLite 看板,資料與排程狀態都不離開本機。沒有任何雲端 / SaaS 強制依賴。本地 LLM (vLLM / Ollama / LM Studio) 是主要運行目標。

Autonomy-first

Dashboard 是觀察台,不是控制台。Agent 可以跑數小時數天,operator 從 kanban + 事件流回頭看怎麼回事;緊急 abort 才介入。Hint 機制是 operator → agent 單向。

Tool API 契約

對外暴露 8 個語意化 endpoint (list_repos / inspect_repo / submit_coding_task / check_task_status / get_task_result / cancel_task / list_tasks / resolve_human_block),fire-and-check 語意,任何上游 LLM client 都能 function-calling 接進來。

Docker 自架

單一 docker compose up 帶起 dashboard + 內建 opencode CLI。Phase 2 雙容器 (opencode 抽離) 已分析過設計,延後到 production-hardening 階段。

四個可替換 runner

echo (測試) / opencode (LiteLLM,真正 local LLM 主力) / anthropic_api (直連 /v1/messages) / claude_cli (cloud)。WORKFLOW.md 一行切換,或 dashboard 🔧 Runner 即時換 (Phase D3)。

三個沒踩到的雷

① tracker 沒寫死 SaaS · ② orchestrator 不會反過來寫 tracker · ③ 內部 FSM 跟 tracker state 嚴格分開。這三條讓 Symphony 可以平移到任何 tracker 後端。

2.2 內部 FSM

每一個 issue 在 orchestrator 內維護一個 RunAttempt,跑下面這個 FSM。注意這跟 tracker 自己的 issue state (Todo / In Progress / Done) 嚴格分開 — orchestrator 從不寫回 tracker。

State意義
UNCLAIMED已知 issue,還沒派出 worker
CLAIMED已起 worker thread,還沒收到第一個 event
RUNNINGworker thread 正在收 runner events
RETRY_QUEUED執行失敗,等待退避時間後重試
RELEASED終態。帶 terminal_reason (handoff / agent_finished / max_turns / stall_timeout / error / aborted / needs_human / tracker_terminal / user_input_required)

2.3 Issue → Tool API 共用的 label 約定

很多 cross-cutting concern 都用 label 編碼而不是新欄位,讓 tracker 不用改 schema 也能存:

Label prefix用途Phase
tool-apiTool API 建立的 issue 標記A
mode:plan|build|reviewper-task runner-level tool whitelistB
trace:<32hex>W3C TraceContext,從 traceparent header 抽出C
parent:<task_id>subtask 指向 parentD1

decompose-pending:1需要 post-finalize hook 拆解D2b

2.4 持久化策略

orchestrator 的 in-memory state 是 source of truth (SPEC §3)。Dashboard SQLite (dashboard.db) 只持久化 5 件事:

  • event_records — agent event 歷史,Replay tab 用
  • hints — operator 補充的 prompt 內容
  • priority_overrides — 拖拉重排的 ranking
  • attempt_history — RELEASED 的 attempt snapshot
  • attempts_state — 行程重啟後 hydrate in-flight 用

刪 dashboard.db 不會影響 orchestrator 的排程正確性 — 只會丟掉 Replay 歷史跟未消費的 hint。

2.5 WebSocket 推播模型 (Phase D 之後)

前端 不再 REST poll /api/v1/state。WS handler 在 connect 後立刻推一份 state_snapshot,然後每個 state-mutating event (fsm_transition / config_changed / workflow_reloaded) 後再 push 一份。

WS filter 支援兩種:?filter=issue:<id1>,<id2> 或 ?filter=project:<id>(Phase E4)。

2.6 安全 / 隔離邊界

  • Bearer auth — REST + WebSocket 共用 DASHBOARD_API_KEY;設了就一律檢查。 /metrics 故意不檢 auth (Prometheus scrape convention)。
  • Workspace 隔離 — 每個 issue 的 worker 在獨立的 workspace 目錄裡跑 (SPEC §7 invariant #1)。
  • Prompt injection 防護 Phase 1 — system-message preamble + 把 untrusted issue/hint 欄位包進 <<<ISSUE_DATA_BEGIN/END>>> 與 <<<OPERATOR_HINT_BEGIN/END>>> 邊界;攻擊者塞偽造邊界 token 會被 defang。
  • Per-task mode — plan = read+grep+bash;build = full toolset;review = read+grep only。Runner argv 層硬性 enforce,不是 prompt suggestion。
  • Rate limit — SYMPHONY_SUBMIT_RATE_LIMIT_PER_MINUTE 限制 submit_coding_task 速率,超過回 429 + Retry-After: 60。

3.2 拆解工作流 (Phase D1 / D2a / D2b)

  • [x]D1 — Subtask graphparent_task_id + depends_on 編碼成 label / blocked_by;parent 失敗時 children 在 dispatch gate 凍結。
  • [x]D2a — Path A (caller-supplied)caller 直接送 subtasks=[...],server 一次建 1 parent + N child;sibling index 翻成真 task_id,單一 idempotency_key 覆蓋整批。
  • [x]D2b — Path B (Symphony-side)decompose=true 讓 Symphony 自己跑 plan-mode runner;post-finalize hook parse 它最後的 assistant message → JSON 子任務陣列 → fan-out children。Fenced code block 內的 marker 被忽略。kill switch SYMPHONY_DECOMPOSE_ENABLED=0。

3.3 升級到人類 (Phase D3)

Agent 在最後 turn 丟出 [HUMAN_REQUIRED] <一句話原因> → orchestrator 把 attempt 收成 terminal_reason=needs_human → 在 kanban / Tool API 都顯示 blocked_for_human → operator 透過 resolve_human_block 寫一段解決方案 hint + force-retry → 下一輪 prompt 自動帶上 hint。

Prompt-injection 防護: fenced code block (```...``` / ~~~...~~~) 內的 marker 不會觸發 escalation,所以 agent 引用 source / docs 不會誤觸。

3.5 Observability — Prometheus + W3C TraceContext

GET /metrics (Prometheus text format,純標準函式庫零依賴):

  • symphony_attempts{state=*} — 即時 attempt by RunState
  • symphony_attempts_finalised_total{terminal_reason=*} — 終態統計
  • symphony_events_total · symphony_events_by_kind_total{kind=*}
  • symphony_hints_total · symphony_hints_pending
  • symphony_priority_overrides · symphony_attempts_state_rows · symphony_idempotency_keys
  • symphony_attempts_blocked_for_human · symphony_human_resolutions_total(Phase D3)

W3C trace_id 透過 traceparent HTTP header 進入 submit_coding_task,server 寫成 trace:<32hex> issue label + 在 response body 回給上游 → 跨行程 log correlation。

4.2 5 欄 Kanban

Pending / Claimed / Running / Retry-Queued / Released。Pending 欄支援拖拉重排(寫入 priority_overrides 表)。每張 IssueCard 顯示 id / title / priority / labels / turns / cost;在「All projects」模式額外顯示 📁 project chip。

4.3 IssueDrawer (點卡片開啟)

  • Overview — 摘要 + pause/resume/abort/retry 動作
  • Events — 完整 agent event 串流(訊息 / 工具呼叫 / 工具結果)
  • Prompt — 渲染好的 prompt 預覽(含 hint 注入)
  • Hints — 操作員留言;resolve_human_block 寫的 hint 也在這裡
  • Workspace — 工作目錄檔案瀏覽 + 前 200 行預覽
  • Replay — 從 SQLite 重放某個 attempt 的事件

4.4 ChatPanel 三個階段

  1. idle — 顯示 "Recent in this project" 歷史 + textarea;若有上一批 batch,顯示 live 進度頁尾(pending/running/done/failed/blocked-for-human 計數,點 "View parent on kanban →" 跳轉)
  2. asking-llm — 顯示 "Asking the LLM to decompose…"
  3. preview — 顯示拆好的子任務,每筆可勾選/取消;另附 raw LLM 回應 <details> 可展開
