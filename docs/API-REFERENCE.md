# Symphony API Reference

> 中文版: [API-REFERENCE.zh-TW.md](API-REFERENCE.zh-TW.md)
> Live OpenAPI spec: **`GET /openapi.json`** (auto-generated by FastAPI)
> Live API explorer: **`GET /docs`** (Swagger UI, auto-generated)

This page is a curated index of every HTTP endpoint Symphony exposes —
organised by audience, not by URL. Each endpoint lists method, path,
required auth, a short purpose, and a curl example. For the authoritative
schema (request bodies, response shapes), open `<host>/openapi.json`.

---

## 0. Conventions

- **Base URL**: `http://localhost:17957` (Docker host) or `http://localhost:7957` (native dev).
- **Auth**: When `DASHBOARD_API_KEY` env is set on the server, every request must include `Authorization: Bearer <token>` on REST and WebSocket alike. Empty key = open mode for local dev only.
- **Content-Type**: All POST/PUT/PATCH bodies are `application/json`.
- **Errors**: Non-2xx responses return `{"detail": "..."}` (FastAPI's default) or `{"detail": "...", "kept_previous": true}` for the workflow validation path.

## 1. Coding Service Tool API (external — for upstream LLM agents)

**Audience**: external LLM agents driving Symphony as a coding-service
backend via function-calling. Fire-and-check semantics — never wait
synchronously on `submit_coding_task`. See
[user-manual.md §5](guide/user-manual.md) for the full integration story
and [coding-service-tool-api.md](design/coding-service-tool-api.md) for
the design rationale.

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/v1/tools/list_repos` | List repos this service can operate on |
| `POST` | `/api/v1/tools/inspect_repo` | Read AGENTS.md + tree + language breakdown (cheap; no agent spawned) |
| `POST` | `/api/v1/tools/submit_coding_task` | Submit a coding task; returns `task_id` < 1s |
| `POST` | `/api/v1/tools/check_task_status` | Poll status + coarse stage |
| `POST` | `/api/v1/tools/get_task_result` | Fetch structured result of a completed task |
| `POST` | `/api/v1/tools/cancel_task` | Abort a running task (idempotent) |
| `POST` | `/api/v1/tools/list_tasks` | Enumerate Tool API tasks with optional `parent_task_id` + `status` filter |

### Example round-trip

```bash
# 1. List repos
curl -X POST http://localhost:17957/api/v1/tools/list_repos \
  -H "content-type: application/json" -d '{}'
# → {"repos": [{"id": "memory", "name": "symphony in-memory tracker", ...}]}

# 2. Submit a task (returns immediately)
curl -X POST http://localhost:17957/api/v1/tools/submit_coding_task \
  -H "content-type: application/json" -d '{
    "task": "Add a /health endpoint returning {status:\"ok\"}",
    "repo": "memory",
    "mode": "build",
    "files_hint": ["src/routes/"],
    "idempotency_key": "client-abc"
  }'
# → {"task_id": "tsk_e3f1...", "status": "pending", "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736"}

# 3. Poll status
curl -X POST http://localhost:17957/api/v1/tools/check_task_status \
  -H "content-type: application/json" -d '{"task_id": "tsk_e3f1..."}'

# 4. When done, fetch structured result
curl -X POST http://localhost:17957/api/v1/tools/get_task_result \
  -H "content-type: application/json" -d '{"task_id": "tsk_e3f1..."}'
```

For a runnable end-to-end demo using only the stdlib see
[`examples/tool_api_client.py`](../examples/tool_api_client.py).

### Key request/response fields

- **`mode`** (`plan` / `build` / `review`): hard tool whitelist enforced
  at the runner level. `plan` = read+grep+bash; `build` = full toolset
  (default); `review` = read+grep only.
- **`idempotency_key`**: 1-128 chars, opaque to Symphony. Same key within
  1h returns the original `task_id` instead of creating a duplicate task.
- **Rate limit** (optional, off by default): set
  `SYMPHONY_SUBMIT_RATE_LIMIT_PER_MINUTE=N` on the server to cap submits.
  Excess submits return **`429 Too Many Requests`** with `Retry-After: 60`.
  Idempotency replays bypass the limit (short-circuit fires first).
- **`trace_id`** (response, always present): 32-hex W3C trace_id bound to
  the task. If the caller sends a W3C `traceparent` HTTP header
  (`00-<32hex>-<16hex>-<2hex>`), Symphony extracts the trace_id portion
  and echoes it back; otherwise a fresh id is generated. Idempotency
  replays return the original task's trace_id so log correlation keeps
  pointing at the same task across retries.
- **`parent_task_id`** + **`depends_on`** (request, optional): Phase D1
  subtask graph. `parent_task_id` is encoded as a `parent:<id>` label on
  the Issue; if the parent's attempt ends in a failure terminal_reason
  (ERROR / ABORTED / MAX_TURNS / STALL_TIMEOUT / USER_INPUT_REQUIRED)
  the orchestrator gates dispatch for every child still pending under
  it. `depends_on` is a list of sibling task_ids that must reach a
  terminal state before this task is dispatched — maps directly to
  `Issue.blocked_by` and is consumed by the existing dispatch gate.
- **`list_tasks`**: returns a flat list of Tool API tasks sorted by
  `created_at` ascending. Filter with `parent_task_id` (walk a child
  graph) or `status` (scan failed-only / running-only). `limit` caps
  the response (default 100, max 500). Read-only — does not spawn an
  agent.
- **`status`** (in `check_task_status`): `pending` / `running` / `done` /
  `failed` / `cancelled`.
- **`stage`** (in `check_task_status`): `queued` / `exploring_codebase` /
  `modifying_files` / `running_tests` / `summarising` / `done` /
  `failed` / `cancelled` — translated from the agent event stream.

## 2. Dashboard / Operator API (internal — for the React SPA)

**Audience**: the React kanban UI, and operators who want to script
human-in-the-loop actions. Most of these endpoints back a single UI
control on the dashboard.

### State + read

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/v1/state` | Full state snapshot for the 5-column kanban |
| `GET` | `/api/v1/issues/{issue_id}` | Single issue detail (hints + rendered prompt preview + workspace path) |
| `GET` | `/api/v1/issues/{issue_id}/events` | Historical agent events for an attempt (with optional `?attempt_number=N&limit=M`) |
| `GET` | `/api/v1/issues/{issue_id}/attempts` | Full attempt history for one issue |
| `GET` | `/api/v1/issues/{issue_id}/workspace` | List files in the per-issue workspace |
| `GET` | `/api/v1/issues/{issue_id}/workspace/file` | Preview the first ~200 lines of a workspace file (`?path=`) |
| `GET` | `/api/v1/workflow` | Current parsed `WORKFLOW.md` config |

### Per-issue actions

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/v1/issues/{issue_id}/hint` | Append an operator note to the next attempt's prompt |
| `POST` | `/api/v1/issues/{issue_id}/pause` | Cooperatively pause a running attempt |
| `POST` | `/api/v1/issues/{issue_id}/resume` | Resume a paused attempt |
| `POST` | `/api/v1/issues/{issue_id}/abort` | Hard cancel; marks RELEASED with `aborted` reason |
| `POST` | `/api/v1/issues/{issue_id}/retry` | Re-dispatch a previously RELEASED issue |

### Global actions

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/v1/emergency_stop` | TopBar kill switch — abort every active worker (idempotent) |
| `POST` | `/api/v1/priority` | Drag-to-reorder the Pending column (priority overrides, not tracker writes) |
| `PATCH` | `/api/v1/config` | Whitelist runtime config: `max_concurrent_agents`, `polling_interval_ms` |
| `PUT` | `/api/v1/workflow` | Hot-reload `WORKFLOW.md` (returns 422 + `kept_previous: true` on parse failure) |

### Live event feed (WebSocket)

```
GET /api/v1/events?filter=issue:MT-1,MT-3
Upgrade: websocket
```

On connect, the server pushes the most recent 100 events for every
non-terminal attempt, then streams live `agent_event` / `fsm_transition` /
`config_changed` / `workflow_reloaded` messages.

Filter syntax: `?filter=issue:<id>,...` narrows to specific issues; omit
the param to receive all.

## 3. Infrastructure endpoints

| Method | Path | Purpose | Auth |
|---|---|---|---|
| `GET` | `/healthz` | Liveness probe; returns `{"ok": true}` | ❌ no auth (intentional — lets `docker compose` healthcheck poll without a token) |
| `GET` | `/metrics` | Prometheus exposition format — 9 metrics covering attempts / events / hints / idempotency. Scrape interval suggestion: 15s | ❌ no auth (Prometheus convention — protect at the reverse-proxy layer) |
| `GET` | `/openapi.json` | OpenAPI 3.1 spec, auto-generated by FastAPI | ❌ no auth |
| `GET` | `/docs` | Swagger UI (interactive API explorer) | ❌ no auth |
| `GET` | `/` | React SPA index | depends on auth config |
| `GET` | `/assets/*` | Static SPA assets (bundled JS/CSS) | ❌ no auth |
| `GET` | `/{full_path}` | SPA fallback (any unmatched path → `index.html`) | depends |

## 4. Auth recipes

```bash
# 1. Generate a token
TOKEN=$(openssl rand -hex 32)
echo "DASHBOARD_API_KEY=$TOKEN" >> .env
docker compose restart dashboard

# 2. Use it on REST calls
curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:17957/api/v1/state

# 3. Use it on WebSocket — most clients accept either header or query
# string; Symphony reads the header on the upgrade request.
wscat -H "Authorization: Bearer $TOKEN" \
  -c ws://localhost:17957/api/v1/events
```
